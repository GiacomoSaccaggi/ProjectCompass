High-Level Structure

ProjectCompass uses a Flask Blueprint architecture with an application factory pattern.

Application Factory – app.py

create_app() initialises Flask, loads configuration from environment, and registers 5 blueprints: auth, catalog, data, agent, and api.

Blueprints – blueprints/

auth.py (login/logout with werkzeug hashing), catalog.py (analysis CRUD, search, pagination), data.py (query runner, uploads), agent.py (LLM chat with sandboxed execution), api.py (REST JSON endpoints).

Business Logic – basefun.py + utils/

The ProjectCompass class aggregates three utility classes via multiple inheritance: DataUtils, AnalysisUtils, and HTMLUtils. Business logic (reading/writing analyses, executing SQL, generating HTML partials) lives here.

View – templates/

Jinja2 HTML templates. A shared header.html partial is parsed at startup and injected into every page via substitute_html(), providing the top bar, sidebar, and CSS/JS includes.

Storage – File System

No relational database. Analyses are stored as folders under Analyses/, each containing metadata.yaml, output_versions.yaml, and subdirectories for code, outputs, and SQL extractions. Data files live in Saved_data/ and queries in Saved_queries/.

Class Inheritance Chain

object
  └── HTMLUtils          (reads header.html, constants.yaml, loads env info)
        └── DataUtils    (DuckDB SQL engine, file upload, data listing)
              └── AnalysisUtils  (read/write analyses, metadata, archive)
                    └── ProjectCompass  (project_folder, create_job_request)

Each class calls super().__init__(dir_path=dir_path, **kwargs), so the chain initialises cleanly from a single dir_path argument.

Request Lifecycle

Browser Request
  → @app.before_request   (session auth check, optional git pull)
  → Flask route function  (app.py)
      → webapp.substitute_html()   (inject header partials)
      → webapp.read_multiple_analysis() / webapp.sql() / etc.
  → render_template()     (Jinja2 fills variables into HTML template)
  → @app.after_request    (optional git add/commit/push for POST)
  → HTTP Response

LLM Chat Layer

The blueprints/agent.py blueprint provides an AI chat interface at /chat. Ollama is lazy-loaded — the application starts and works fully without it. User questions are converted to pandas expressions by the LLM, then executed via safe_execute_pandas() which uses AST validation to block import, open(), exec(), and eval().
